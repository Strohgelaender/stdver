Standard Versioning Scheme
The Standard Versioning is defined by the following version identifier scheme:
M.NpR [.D] [+H] [-S]
  • M: /\d+/ (M ≥ 0) — Major Version
    Major version of solution. Bumped on major technical or domain-specific changes only. A bump resets the Minor Version N and Revision R, too. On upgrades, human interaction is expected.
  • N: /\d+/ (N ≥ 0) — Minor Version
    Minor version of the solution within the Major Version. Bumped on new features. A bump resets the Revision R, too. Backward compatibility is seeked for, but not assured. On upgrades, human interaction might be required.
  • p: /(a|b|rc|\.)/ — Maturity Level
    Maturity level within Major Version M and Minor Version N.
    • a: Alpha
      Early version of the solution with incomplete and unstable functionalities, to get feedback on product. For Alpha maturity levels, the Revision R usually starts with 1.
    • b: Beta
      Early version of the solution with complete but still unstable functionalities, to stabilize product. For Alpha maturity levels, the Revision R usually starts with 1.
    • rc: Release Candidate
      Mature version of the solution with complete and stable functionalities, to catch last-minute problems just before release. For Alpha maturity levels, the Revision R usually starts with 1.
    • .: Release
      Mature version of the solution with complete and stable functionalities, available for full production use. For Alpha maturity levels, the Revision R usually starts with 0.
  • R: /\d+/ (R ≥ 0) — Revision
    Revision of the Maturity Level within Major Version M and Minor Version N. Bumped for every Maturity Level m. On Release Update Maturity Levels, full backward compatibility is assured. On upgrades no human interaction is required.
  • D: /\d{8}/ — Release Date
    The optional release date of year, month and date, represented in sort-supporting ISO notation.
  • H: /[\dA-F]{4}/ — Source Hash
    The optional 16-bit hashing checksum of the source code of the solution, represented as a upper-case hexadecimal value. This usually is a SHA1 or MD5 hash (8 bytes, 128 bit), folded 3 times through an XOR-operation into 2 bytes / 16 bit.
  • S: /(XA|LA|EA|GA)/ — Availability Scope
    The optional availability scopes of the release.
    • XA: No Availability
      No public availability of solution at all. The scope for all Development and sometimes Snapshot point-in-times.
    • LA: Limited Availability
      Limited public availability of solution. Usually for releases after the End-of-Life-Announcement (EOLA) or for releases with specific customer features.
    • EA: Early Availability
      Early public availability of solution for early market. Usually for Beta or Release Candidate levels or for Release and initial Release Update levels.
    • GA: General Availability
      Late public availability of solution for mainstream market. Usually for Release and sometimes just for Release Update levels.
Version Examples
The following are samples of versions based on Standard Versioning.
  • 1.2.3:
    a release version.
  • 1.2.3-GA:
    a release version with an explicit availability scope.
  • 1.2b3.20230821-LA:
    a snapshot of a beta version, made on a particular date, with explicit availability scope.
  • 1.2a3.20230821+42FA-XA:
    a snapshot of an alpha version, made on a particular date, with explicit source hash and availability scope.
  • 1.2a3+42FA-XA:
    a snapshot of an alpha version, with explicit source hash and availability scope.
About
The Standard Versioning dates back to early definitions in the year 2008 by Dr. Ralf S. Engelschall in the context of its Software Architecture trainings. It was formally defined and publically published on stdver.org in 2023 to have a convenient reference at hand on the Web. It is motivated by the usual maturity-based intution one has in mind when when it comes to solution versions.
